Static vs Virtual / Override

Why they conflict, how polymorphism works, and when to use new instead.

The Conflict Polymorphism Fix Keeping Static Side-by-Side Key Takeaways
01

The static + virtual/override Conflict

In C#, static methods cannot be virtual or overridden. The two keywords represent fundamentally opposite mechanisms:

⚡ virtual Runtime

The method can be overridden in derived classes. The decision of which version to call is made at runtime based on the actual object type — this is polymorphism.

🔒 static Compile-time

The method belongs to the class, not an instance. It is resolved at compile time based on the variable's declared type — no polymorphism possible.

🚫
Compiler Error — combining static with virtual or override will not compile:
"Static member 'Base.Print()' cannot be marked as override, virtual, or abstract."
BrokenCode.cs — will NOT compile
class Base
{
    // ❌ static + virtual is illegal
    public static virtual void Print()
    {
        Console.WriteLine("Base");
    }
}

class Derived : Base
{
    // ❌ static + override is illegal
    public static override void Print()
    {
        Console.WriteLine("Derived");
    }
}
02

The Right Way — virtual / override for Polymorphism

If your goal is runtime polymorphism — where the method that runs depends on the actual object type, not the variable type — remove static entirely and use instance methods:

Polymorphism.cs — runtime binding
class Base
{
    public virtual void Print()         // can be overridden
    {
        Console.WriteLine("Base");
    }
}

class Derived : Base
{
    public override void Print()        // replaces Base.Print()
    {
        Console.WriteLine("Derived");
    }
}

// Variable type = Base, actual object type = Derived
Base obj = new Derived();
obj.Print();  // runtime decides → calls Derived.Print()
Output Derived
✅
Even though obj is declared as type Base, the runtime looks at the actual object (which is a Derived) and calls Derived.Print(). This is runtime polymorphism — the whole point of virtual/override.
03

Keeping static — Method Hiding with new

If you must keep static, you cannot override. You can instead hide the base method using the new keyword — but this behaves completely differently from polymorphism:

StaticHiding.cs — compile-time binding
class Base
{
    public static void Print()
    {
        Console.WriteLine("Base Static");
    }
}

class Derived : Base
{
    public new static void Print()     // hides Base.Print()
    {
        Console.WriteLine("Derived Static");
    }
}

Base obj = new Derived();
obj.Print();                           // compiled as Base.Print()

Derived.Print();                        // explicitly calls Derived
obj.Print() Base Static Derived.Print() Derived Static
💡
Why does obj.Print() print "Base Static"?
Because obj is declared as type Base. Static methods are resolved at compile time using the declared type of the variable, not the actual runtime object. The compiler locks in Base.Print() before the program even runs.

05

Key Takeaways

✅ What to Remember

  • static + virtual/override is illegal. The compiler rejects it outright — they are fundamentally incompatible.
  • Use virtual/override for polymorphism. The runtime decides which method runs based on the actual object type, not the variable type.
  • Use static + new for method hiding. The compiler decides at compile time based on the declared variable type. No runtime dispatch.
  • new hides, it does not override. The base method still exists and is still called when using a base-typed variable.

Key Insight

The core question is: who decides which method runs — the compiler or the runtime?

virtual/override → the runtime decides (actual object type wins).
static/new → the compiler decides (declared variable type wins).

When you want subclasses to substitute their own behavior seamlessly, always choose virtual/override. Reserve static for utility methods that never need polymorphic dispatch.
