Object-Oriented Programming in C#

C# is a modern, object-oriented programming language that organizes programs around objects and data rather than actions and logic. OOP makes large software systems more modular, reusable, and maintainable by modeling real-world entities as objects.

What is OOP?

Object-oriented programming (OOP) is a paradigm based on the concept of "objects," which can contain data (fields/properties) and code (methods/functions). OOP structures software design around data and the associated behavior.

The Four Pillars of OOP

1. Encapsulation

Encapsulation means bundling the data (properties) and the operations (methods) that work on the data into a single unit called a class. It protects data by restricting external access and exposing only what is necessary.

Usage: Use private, public, protected, and other access modifiers to control access.

public class Person
{
    private string name; // encapsulated field

    public string Name // property for controlled access
    {
        get { return name; }
        set { name = value; }
    }
}

2. Inheritance

Inheritance allows one class (child/derived) to inherit fields and methods from another class (base/parent), promoting code reuse and extensibility.

Usage: Use the : symbol for inheritance.

public class Animal
{
    public void Eat() { /*...*/ }
}

public class Dog : Animal
{
    public void Bark() { /*...*/ }
}
// Dog has both Eat() and Bark()

3. Abstraction

Abstraction simplifies complex reality by modeling classes appropriate to the problem and working at the most relevant level for a particular aspect. It hides unnecessary details from end-users.

Usage: Use abstract classes or interfaces for abstraction.

public abstract class Shape
{
    public abstract double Area();
}

4. Polymorphism

Polymorphism allows one interface to be used for a general class of actions. The specific action is determined by the exact nature of the situation.

Usage: Implement with method overloading/overriding and interfaces.

public class Animal
{
    public virtual void Speak() { Console.WriteLine("Some sound"); }
}

public class Dog : Animal
{
    public override void Speak() { Console.WriteLine("Bark"); }
}

Real-World Example

// Base class
public class Account
{
    private double balance;
    public Account(double initialBalance) { balance = initialBalance; }
    public double Balance { get { return balance; } }
    public void Deposit(double amount) { if (amount > 0) balance += amount; }
    public void Withdraw(double amount) { if (amount > 0 && amount <= balance) balance -= amount; }
}

// Derived class with polymorphism and extension
public class SavingsAccount : Account
{
    private double interestRate;
    public SavingsAccount(double initialBalance, double rate) : base(initialBalance) { interestRate = rate; }
    public void AddInterest() { Deposit(Balance * interestRate); }
}

Advantages of Using OOP in C#

  • Code Reusability: Inheritance encourages reuse and reduces duplication.
  • Modularity: Easier maintenance and debugging due to separation into classes.
  • Scalability: Easy to extend software by creating new subclasses.
  • Security and Flexibility: Encapsulation and abstraction hide complexity and internal details.

Encapsulation

Encapsulation is one of the fundamental principles of OOP. It refers to bundling data (fields/properties) and the methods that operate on that data within a single unit (class), while controlling access to that data.

Key Benefits

  • Protects object integrity by preventing external code from directly modifying internal state
  • Provides controlled access through public methods and properties
  • Allows internal implementation changes without affecting external code
  • Enhances security and data hiding

Implementation with Properties

public class BankAccount
{
    // Private field - encapsulated data
    private decimal _balance;

    // Public property - controlled access
    public decimal Balance
    {
        get { return _balance; }
        private set { _balance = value; }
    }

    // Public methods to modify state
    public void Deposit(decimal amount)
    {
        if (amount > 0)
            _balance += amount;
    }

    public bool Withdraw(decimal amount)
    {
        if (amount > 0 && amount <= _balance)
        {
            _balance -= amount;
            return true;
        }
        return false;
    }
}

Access Modifiers for Encapsulation

C# provides several access modifiers to control visibility:

  • private - Only accessible within the same class
  • public - Accessible from anywhere
  • protected - Accessible within the class and derived classes
  • internal - Accessible within the same assembly

Best Practice

Always start with the most restrictive access level (usually private) and only increase visibility when necessary. This principle is known as "information hiding" and is central to good encapsulation.

Inheritance in C#

Inheritance allows a class (derived/child) to acquire members (fields, methods, properties) from another class (base/parent). C# supports several inheritance types, though some are limited to interfaces due to language design.

Types of Inheritance

1. Single Inheritance

A derived class inherits from exactly one base class. This is the most fundamental form supported in C#.

class Animal {
    public void Eat() { Console.WriteLine("The animal eats."); }
}

class Dog : Animal {
    public void Bark() { Console.WriteLine("The dog barks."); }
}
// Usage: 
Dog d = new Dog();
d.Eat();  // Output: The animal eats.
d.Bark(); // Output: The dog barks.

2. Multilevel Inheritance

A chain of inheritance: a derived class inherits from a base class, and a further class inherits from that derived class.

class Grandparent {
    public void Show() { Console.WriteLine("Grandparent class."); }
}

class Parent : Grandparent {
    public void Display() { Console.WriteLine("Parent class."); }
}

class Child : Parent {
    public void Print() { Console.WriteLine("Child class."); }
}
// Usage:
Child c = new Child();
c.Show();    // Grandparent class.
c.Display(); // Parent class.
c.Print();   // Child class.

3. Hierarchical Inheritance

Multiple classes inherit from a single base class. This is useful for common base functionality.

class Shape {
    public void Draw() { Console.WriteLine("Drawing shape."); }
}

class Circle : Shape {
    public void CircleDraw() { Console.WriteLine("Drawing circle."); }
}

class Rectangle : Shape {
    public void RectangleDraw() { Console.WriteLine("Drawing rectangle."); }
}

4. Multiple Inheritance (Using Interfaces Only)

A class can inherit from multiple interfaces (but not multiple classes). Enables composition of multiple contracts.

interface IA {
    void A();
}
interface IB {
    void B();
}

class C : IA, IB {
    public void A() { Console.WriteLine("A"); }
    public void B() { Console.WriteLine("B"); }
}

5. Hybrid Inheritance (Achievable via Interfaces)

A combination of two or more types of inheritance, such as hierarchical and multiple, generally realized via interfaces in C#.

Important Note

C# does NOT support multiple inheritance of classes directly to prevent ambiguity but fully supports it with interfaces. Hybrid inheritance is possible only via a combination of classes and interfaces.

Polymorphism in C#

Polymorphism is a fundamental concept in object-oriented programming (OOP) that allows objects to be treated as instances of their parent class rather than their actual class. In simple terms, polymorphism means "many forms"—the same method or operation behaves differently on different classes or data types.

C# supports two primary types of polymorphism:

  • Compile-time polymorphism (static polymorphism): Achieved by method overloading or operator overloading.
  • Run-time polymorphism (dynamic polymorphism): Achieved by method overriding using inheritance and the virtual/override keywords.

1. Compile-Time Polymorphism (Method Overloading)

Method overloading allows multiple methods in the same class to have the same name, provided they have different parameter lists. The correct method is picked at compile-time based on the arguments passed.

using System;

class Printer
{
    public void Print(int number)
    {
        Console.WriteLine("Printing int: " + number);
    }

    public void Print(string text)
    {
        Console.WriteLine("Printing string: " + text);
    }
}

class Program
{
    static void Main()
    {
        Printer p = new Printer();
        p.Print(10);            // Output: Printing int: 10
        p.Print("Hello C#");    // Output: Printing string: Hello C#
    }
}

Here, two Print methods are defined—one accepts an int, the other a string. The method selected depends on the argument's type at compile time.

2. Run-Time Polymorphism (Method Overriding)

Run-time polymorphism uses inheritance: a base class declares a virtual method, and derived classes override it with the override keyword. Which method gets called is determined at runtime based on the object's actual type.

using System;

class Animal
{
    public virtual void Speak()
    {
        Console.WriteLine("The animal makes a sound.");
    }
}

class Dog : Animal
{
    public override void Speak()
    {
        Console.WriteLine("Dog barks: Woof!");
    }
}

class Cat : Animal
{
    public override void Speak()
    {
        Console.WriteLine("Cat meows: Meow!");
    }
}

class Program
{
    static void Main()
    {
        Animal myDog = new Dog();
        Animal myCat = new Cat();

        myDog.Speak();     // Output: Dog barks: Woof!
        myCat.Speak();     // Output: Cat meows: Meow!
    }
}

The Speak() method is defined in the base class and overridden in each derived class. The method that is executed is determined at runtime using the actual object type (either Dog or Cat).

3. Polymorphism with Interfaces

C# also supports polymorphism using interfaces, where different classes implement the same interface but with varying behavior.

public interface IAnimal
{
    void MakeSound();
}

public class Dog : IAnimal
{
    public void MakeSound()
    {
        Console.WriteLine("Dog barks: Woof!");
    }
}

public class Cat : IAnimal
{
    public void MakeSound()
    {
        Console.WriteLine("Cat meows: Meow!");
    }
}

class Program
{
    static void Main()
    {
        IAnimal animal;

        animal = new Dog();
        animal.MakeSound(); // Output: Dog barks: Woof!

        animal = new Cat();
        animal.MakeSound(); // Output: Cat meows: Meow!
    }
}

Both Dog and Cat implement the IAnimal interface. At runtime, the MakeSound() method executes the appropriate version based on the object's actual type.

Why Use Polymorphism?

  • Code Reusability: Write code using the base type; works across all derived types.
  • Extensibility: Easily add new classes with minimal changes to existing code.
  • Maintainability: Common interface for diverse implementations simplifies management.

Abstraction in C#

Abstraction in C# is an OOP principle that allows you to hide complex implementation details and show only the essential features of an object or system. It helps manage complexity by exposing a cleaner, more user-friendly interface, letting users interact with objects without needing to understand their inner workings.

Why Use Abstraction?

  • Simplifies complex systems by exposing only what is necessary.
  • Improves flexibility and security by shielding internal details.
  • Allows different implementations underlying the same abstraction.

How to Achieve Abstraction in C#

C# supports abstraction mainly in two ways:

  • Abstract Classes and Methods
  • Interfaces (for complete abstraction)

1. Abstraction with Abstract Classes

Abstract classes allow you to declare methods without implementing them. Derived (sub-)classes must provide the concrete implementation.

// Abstract class
abstract class Animal
{
    // Abstract method (no implementation)
    public abstract void MakeSound();

    // Regular method (with body)
    public void Sleep()
    {
        Console.WriteLine("Zzz");
    }
}

// Derived class (inherits from Animal)
class Dog : Animal
{
    public override void MakeSound()
    {
        Console.WriteLine("Dog says: Woof!");
    }
}

// Usage
class Program
{
    static void Main()
    {
        Dog myDog = new Dog();
        myDog.MakeSound(); // Output: Dog says: Woof!
        myDog.Sleep();     // Output: Zzz
    }
}

Description:

  • The Animal abstract class defines a method MakeSound() with no implementation (the abstraction).
  • The Dog class provides the concrete details by overriding MakeSound().
  • User only needs to call MakeSound() or Sleep(); they don't need to know how the sound is produced.

2. Abstraction with Interfaces

Interfaces define contracts that must be fulfilled by implementing classes—commonly used for complete abstraction.

public interface IMediaPlayer
{
    void Play();
    void Pause();
    void Stop();
}

public class AudioPlayer : IMediaPlayer
{
    public void Play()  { Console.WriteLine("Playing audio"); }
    public void Pause() { Console.WriteLine("Audio paused"); }
    public void Stop()  { Console.WriteLine("Audio stopped"); }
}

Description:

  • The interface IMediaPlayer exposes only playing functionality, not how audio is decoded or managed internally.
  • The implementation in AudioPlayer is hidden from the user.

Real-World Analogy

When you drive a car, you use the steering wheel, brakes, and accelerator (the abstraction). You don't need to know the mechanics of how the engine works or how the brakes apply—the complexity is hidden.

Key Points

  • Use abstract classes and interfaces for abstraction.
  • Abstract class: Use when you want to provide some shared code and some forced implementation.
  • Interface: Use for total abstraction or when multiple, unrelated classes should agree to the same contract.
  • Abstraction helps in building secure, flexible, and maintainable applications.

Access Modifiers in C#

Access modifiers in C# are keywords that define the visibility and accessibility of types (classes, structs, interfaces) and their members (fields, methods, properties, etc.). Appropriate use of access modifiers enforces encapsulation and regulates which parts of your code can interact with others.

List of Access Modifiers

1. public

Description: No restrictions — accessible from any other code in the same assembly or another assembly that references it.

public class Car {
    public string model = "Tesla";
}

2. private

Description: Accessible only within the containing class or struct. This is the most restrictive access.

class Car {
    private string model = "Tesla";
}

3. protected

Description: Accessible within the containing class and any class derived from it (i.e., subclasses), even if they are in different assemblies.

class Vehicle {
    protected int maxSpeed = 200;
}

4. internal

Description: Accessible only within the same assembly (project). Not visible outside the assembly, even by derived classes.

internal class Helper {
    internal void Process() { }
}

5. protected internal

Description: Accessible within the same assembly, OR from derived classes in other assemblies. This is a combination of protected and internal.

protected internal void Calculate() { }

6. private protected

Description: Accessible only within the containing class or types derived from the containing class, but only within the same assembly.

private protected int Age { get; set; }

7. file (C# 11 and above)

Description: The declared type is only visible in the current source file. Generally used for source generators and partial classes.

file class DiagnosticHelper { }

Summary Table

Modifier Accessible from...
public Any code anywhere (no restriction)
private Containing class or struct only
protected Containing class and its derived types
internal Any code within the same assembly
protected internal Same assembly or derived types in any assembly
private protected Same assembly and derived types of the class
file The same source file only

Best Practices

  • Always use the most restrictive access level that still allows your code to function correctly
  • Default to private for class members unless there's a specific reason to expose them
  • Use public only for the API you want to expose to external consumers
  • Prefer protected over public when inheritance is involved
